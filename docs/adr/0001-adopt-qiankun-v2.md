# ADR-0001: 采用 qiankun v2 稳定线作为微前端底座

- 状态：Accepted
- 日期：2026-03-22
- 决策者：仓库维护者 / 平台负责人

## 背景

目标是为多个既有系统建立一个可生产运行的统一前端入口，重点解决：

- 访问一致性
- 设计风格一致性
- 账号一致性
- 独立部署与渐进迁移

当前仓库为空仓，说明我们拥有较大的架构自由度，也意味着必须先把边界定义清楚。

## 备选方案

### A. qiankun

优点：

- 中文生态成熟，文档直接
- 支持多技术栈微应用接入
- 基于 HTML entry，适合接入存量系统
- 提供沙箱、预加载、手动挂载等能力
- 适合“统一入口 + 渐进迁移”

缺点：

- 不是完整平台，只是运行时框架
- 样式隔离、鉴权、缓存、灰度、监控都需要额外治理
- 生产问题多集中在部署规范与边界治理，而不是 API 本身

### B. single-spa 原生方案

优点：

- 更底层、更灵活
- 对定制平台团队友好

缺点：

- 平台封装成本更高
- 需要自行补更多接入层与治理规范

### C. Module Federation 为核心方案

优点：

- 构建期依赖共享能力强
- 对统一技术栈、统一构建链项目友好

缺点：

- 对存量系统、异构技术栈、独立部署治理并不天然更简单
- 更偏“模块共享与远程组件”，不是完整的微前端治理方案

## 决策

采用 `qiankun v2` 稳定线作为当前仓库的微前端底座。

更具体地说：

- 运行时编排由 qiankun 承担
- 统一入口由 shell 承担
- 统一身份、设计系统、监控、发布治理由仓库内自建平台层承担
- 本期不采用 `v3.0.0-rc.0`

## 关键理由

### 1. 适合存量系统接入

用户场景是“已有多个系统”，这比“从零设计同构应用”更接近 qiankun 的优势区间。

### 2. 兼容渐进迁移

我们不需要一次性把所有系统迁到同一技术栈或同一仓库，可以先完成：

- 主应用搭建
- 一个微应用接入
- 契约和治理能力打通

### 3. 技术方案需要稳定性优先

基于官方发布页，`v2.10.16` 为当前 `Latest`，而 `v3.0.0-rc.0` 仍是预发布版本。因此当前项目应优先稳定，不追逐新 API。

## 明确不做的错误假设

以下说法不成立：

- “上了微前端就自动有统一账号”
- “上了 qiankun 就自动有设计一致性”
- “上了沙箱就不会再有样式冲突”

正确理解是：

- qiankun 解决的是前端运行时编排
- SSO 解决的是账号一致性
- 设计系统解决的是视觉一致性
- 工程规范解决的是部署与回滚一致性

## 结果与影响

### 正向影响

- 支持分阶段接入现有系统
- 支持跨团队独立发布
- 支持主应用统一导航、监控和布局
- 支持异构技术栈并存

### 负向影响

- 平台治理工作会明显增加
- 对部署规范、缓存策略和契约测试要求更高
- 初期需要投入在文档、脚手架、共享 SDK 和观测体系

## 后续动作

- 以 shell + contracts + auth facade + design tokens 作为第一批平台资产
- 先接入 1 个真实微应用验证契约
- 完成后再扩到第二个系统，而不是一开始并行改造全部系统
