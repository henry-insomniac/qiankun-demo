# 实施路线图

## 1. 目标

以最小风险方式，把空仓演进成一个可承接真实业务的 qiankun 平台仓。

## 2. 实施原则

- 先定边界，再写代码
- 先通一条真实链路，再扩规模
- 平台能力优先于样例堆砌
- 每一阶段都可验证、可回滚

## 3. 里程碑

### Phase 0: 文档和治理基线

交付物：

- `AGENTS.md`
- ADR
- 架构蓝图
- 接入契约
- 质量门禁文档
- 路线图

验收标准：

- 新加入的开发者只读文档就能理解边界
- 能明确回答“为什么选 qiankun v2，而不是 v3 预发布”

### Phase 1: Shell 主应用骨架

交付物：

- `apps/shell`
- 基础路由
- 顶部/侧边布局骨架
- 404 / 错误边界
- qiankun 注册中心
- app manifest 读取机制

验收标准：

- 主应用可以独立运行
- 可从配置加载微应用定义
- 可展示微应用占位和错误态

### Phase 2: 第一个真实微应用接入

交付物：

- 1 个真实微应用生命周期改造
- 独立运行与被挂载双模式
- 路由 base 对齐
- publicPath 与资源策略落地

验收标准：

- 能通过 shell 正常进入该应用
- 能直接独立访问该应用
- 刷新深链不 404
- mount/unmount 无泄漏

### Phase 3: 统一身份与设计系统

交付物：

- `packages/auth-sdk`
- `packages/design-tokens`
- shell 与微应用的 theme/user 注入链路
- 全局退出与会话续期

验收标准：

- 登录一次后进入所有接入应用
- 退出一次后所有应用失效
- 设计令牌在 shell 与微应用间一致生效

### Phase 4: 质量门禁与观测

交付物：

- 单元/契约/E2E 测试
- 日志、指标、异常上报
- 版本清单与回滚机制
- 灰度策略

验收标准：

- 微应用加载失败可降级
- 可通过配置回退单个应用版本
- 关键路径有自动化回归

### Phase 5: 第二个微应用接入与平台化复盘

交付物：

- 第二个真实系统接入
- 抽取复用脚手架
- 更新 ADR 和接入模板

验收标准：

- 第二个系统接入成本显著低于第一个
- 文档能沉淀出稳定模板

## 4. 第一批代码建议顺序

推荐从以下顺序开始写代码：

1. `packages/contracts`
2. `apps/shell`
3. `packages/auth-sdk`
4. `packages/design-tokens`
5. 第一个微应用适配层
6. `apps/playground`

原因：

- 先把边界写清楚
- 再把主控面搭起来
- 最后接业务系统，避免边接边改契约

## 5. 任务分解建议

### Track A: 平台骨架

- 仓库工程化
- shell
- manifest
- shared utils

### Track B: 身份接入

- 统一登录方案
- 会话刷新
- 全局退出

### Track C: 第一个微应用改造

- 生命周期导出
- publicPath
- 路由 base
- 样式边界

### Track D: 质量体系

- CI
- 契约测试
- E2E
- 监控

## 6. 风险驱动优先级

最先解决的不是 UI，而是这些高风险项：

1. 登录态一致性
2. publicPath 与资源 404
3. 深链刷新 404
4. 样式污染
5. 回滚能力

## 7. 关键里程碑的完成定义

### M1: 平台可运行

- shell 可启动
- manifest 可解析
- 错误边界存在

### M2: 一个微应用可生产接入

- 独立运行可用
- 被挂载可用
- 登录态贯通
- 资源路径正确

### M3: 平台具备运营能力

- 监控完备
- 可回滚
- 自动化回归存在

## 8. NOT in Scope

本路线图当前明确不包含：

- 全量系统迁移排期
- 多团队组织治理流程
- 商业化主题市场
- 跨端统一容器

## 9. 下一步建议

如果你希望我继续往前推进，最合理的下一步不是继续写更多文档，而是直接开始搭第一版工程骨架：

```text
apps/shell
packages/contracts
packages/auth-sdk
packages/design-tokens
apps/playground
```

然后选一个真实系统做首个接入验证。
